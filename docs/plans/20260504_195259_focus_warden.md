## What This Is

A Linux desktop focus enforcer for Hyprland/Wayland that:
1. Reads a defined set of tasks from Notion at session start
2. Builds a schedule relative to PC-on time
3. Presents a fullscreen blocker before each scheduled work block
4. Forces stretch and gym breaks as scheduled interruptions
5. Warns before a 1AM cutoff and can power off the machine
6. Logs real work and interruption durations
7. Supports an instant pause button through a real control channel

**Important constraint:** v1 is a strong blocker, not a secure lock. It can be annoying and sticky, but it is not a cryptographic or compositor-level session lock.

## Technical Direction

- **Language:** Python 3.11+
- **Packaging:** `pyproject.toml` with `setuptools` or `hatchling`
- **Config:** `config.toml`
- **Notion:** `notion-client`
- **UI:** Tkinter for blocker window in v1
- **Notifications:** `notify-send`
- **Storage:** SQLite in `~/.local/share/focus-warden/warden.db`
- **Runtime model:** single long-running daemon plus local IPC control socket
- **Autostart:** systemd user service
- **Window manager integration:** Hyprland keybind calls control CLI, not internal Python state directly

### Why this direction

- Tkinter is good enough for a personal fullscreen blocker.
- A local daemon with IPC avoids the broken “spawn a second Python process and mutate globals” design.
- Packaging with `pyproject.toml` is cleaner than freezing ad hoc requirements from a shell session.

## Core Design Decisions

### 1. Enforcement model

Use a fullscreen blocker window before a task starts and before breaks start. The blocker requires an explicit action such as:
- typing the task name to begin a task block
- clicking a break-start button for stretch/gym blocks

This is **not** a secure lock. The app should say so in the documentation and code comments.

### 2. Control plane

All external controls must go through IPC:
- `pause`
- `resume`
- `give-up`
- `status`
- `skip-current-break` if enabled

Recommended implementation:
- Unix domain socket in `~/.local/state/focus-warden/control.sock`
- JSON messages with a tiny command schema

### 3. Runtime model

Use a state machine rather than a polling script with hidden mutable globals.

Suggested states:
- `idle`
- `awaiting_task_start`
- `task_active`
- `awaiting_break_start`
- `break_active`
- `paused`
- `given_up`
- `finished`

### 4. Scheduling rules

The schedule is generated once at session start from:
- session start timestamp
- normalized Notion tasks
- fixed break settings
- fixed shutdown deadline

When the user pauses, future items are shifted by the pause duration, except the hard shutdown deadline if that remains absolute.

### 5. Learning

Duration learning is deferred until v1 logging is correct. v2 should learn from normalized task labels or explicit categories, not raw task names alone.

## Data Model

### SQLite location

`~/.local/share/focus-warden/warden.db`

### Tables

```sql
CREATE TABLE sessions (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    started_at TEXT NOT NULL,
    ended_at TEXT,
    shutdown_deadline TEXT NOT NULL,
    status TEXT NOT NULL
);

CREATE TABLE tasks (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    session_id INTEGER NOT NULL,
    notion_task_id TEXT,
    title TEXT NOT NULL,
    normalized_key TEXT NOT NULL,
    scheduled_start TEXT NOT NULL,
    scheduled_duration_minutes INTEGER NOT NULL,
    actual_start TEXT,
    actual_end TEXT,
    actual_minutes REAL,
    status TEXT NOT NULL,
    FOREIGN KEY(session_id) REFERENCES sessions(id)
);

CREATE TABLE interruptions (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    session_id INTEGER NOT NULL,
    kind TEXT NOT NULL,          -- pause, stretch, gym, give_up
    started_at TEXT NOT NULL,
    ended_at TEXT,
    duration_minutes REAL,
    FOREIGN KEY(session_id) REFERENCES sessions(id)
);

CREATE TABLE control_events (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    session_id INTEGER,
    event_type TEXT NOT NULL,
    payload_json TEXT,
    created_at TEXT NOT NULL
);
```

### Notes

- `normalized_key` is for future learning.
- `status` values should be explicit enums in code, not free text sprinkled everywhere.

## Phase 3 — Scheduler

### Scheduling rules

1. Use `session_start` as the single time anchor.
2. Compute `shutdown_deadline` from configured `hard_shutdown_time`.
3. Insert the first task after `task_start_grace_seconds`.
4. Insert stretch breaks after each configured amount of accumulated work.
5. Insert gym as a fixed break block at `session_start + gym_offset_hours`.
6. If a task would cross the gym boundary:
   - either split the task around gym, or
   - delay gym until task end.

Pick one rule and document it. Recommended v1 rule: delay gym until the current task ends.

### Scheduler output

Create `ScheduleItem` objects with:
- `kind`
- `title`
- `scheduled_start`
- `duration_minutes`
- `task_ref`

### Step 3.1 — Pure function

`build_schedule(tasks, session_start, config) -> list[ScheduleItem]`

This module must be pure and testable. No `datetime.now()` calls inside it except in test helpers.

### Step 3.2 — Tests

Add tests for:
- no tasks
- tasks shorter than gym offset
- task crossing gym offset
- repeated stretch insertion
- shutdown warning placement

Deliverable:
- schedule printed and tested from sample input

## Phase 4 — UI Blocker

### V1 blocker behavior

Build a fullscreen topmost window that:
- displays current task or break
- shows scheduled duration and current time
- accepts a typed task confirmation for tasks
- provides a visible pause control
- provides a hidden or delayed give-up control

### Constraints

- It can be annoying and sticky.
- It cannot be treated as a secure lock.
- It should degrade gracefully if the window loses focus.

### Recommended response to focus loss

If the blocker window loses focus:
- re-raise itself
- send a notification
- log a control event

Do not pretend this guarantees enforcement.

## Phase 6 — State Machine and Daemon

### Main runtime rules

Create `src/focus_warden/state_machine.py` and `src/focus_warden/daemon.py`.

The daemon should:
- load config
- initialize DB
- fetch tasks
- build schedule
- start control server
- process schedule items one at a time
- react to pause/resume/give-up commands
- emit notifications

### Pause behavior

On pause:
- record interruption start
- transition to `paused`
- suspend timers

On resume:
- compute pause duration
- shift future schedule items by that duration
- record interruption end
- return to the previous state

### Give-up behavior

For v1:
- require a confirmation phrase plus cooldown
- mark session as `given_up`
- close blocker windows
- stop the daemon for the day

No email flow in v1.

Deliverable:
- daemon runs a complete session with pause/resume/give-up
