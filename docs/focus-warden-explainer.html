Technical build explainer

Locked-In

This page explains how the repo is assembled, not what it promises to do. The build starts at the CLI entry point, loads configuration, fans out into a simple planner runtime and a legacy daemon runtime, and then connects scheduling, UI, SQLite persistence, and a local Unix-socket control plane.

How the build is layered

The code is deliberately split into layers so each part has one job. CLI routing and config loading happen first, the persistence layer comes next, the schedule builder stays pure, and the UI/runtime code gets to handle side effects later.

Bootstrap vs runtime

pyproject.toml and main.py define how the package starts before any session logic exists.

Pure logic vs side effects

scheduler.py computes a plan without touching the world. The daemon, DB, UI, and notifications handle effects.

Default path vs legacy path

The current service launches locked-in run, which hits the simple planner path instead of the daemon.

The most important implementation detail: the service points at locked-in run, so the shipped default behavior is the planner runtime. The daemon is present, but it is not the installed entry point.

Why this shape works

Side effects are isolated The schedule builder can be tested in isolation because it does not depend on Qt, SQLite, or the current wall clock.
State is explicit The daemon uses a small state machine instead of ad hoc globals so pause, resume, and give-up are inspectable transitions.
Local storage is simple Both SQLite stores are schema-first and low ceremony, which keeps the runtime logic readable.

The schedule engine

`scheduler.py` is a pure function. It consumes a task list, a session start, and config values, then emits ordered ScheduleItem objects without touching Qt, SQLite, or the current clock.

Stage 1

Initialize cursor

Start from session_start plus the grace period. Derive shutdown_deadline and the gym anchor up front.

Stage 2

Place task blocks

Walk the normalized task list in order and allocate estimate_minutes or the default task length.

Stage 3

Inject interruptions

Insert stretch breaks whenever accumulated_work crosses the interval threshold. Insert the gym break once when the cursor reaches the configured offset.

Stage 4

Clamp to deadline

If a task would run past the shutdown boundary, clip it or stop scheduling. Then append the warning window and final shutdown item.

Rule summary in code shape

# build_schedule(tasks, session_start, config) cursor = session_start + grace_seconds shutdown_deadline = session_start.replace(hard_shutdown_time) gym_time = session_start + gym_offset_hours for task in tasks: maybe insert stretch break maybe insert gym break once duration = task.estimate_minutes or default_task_minutes if cursor + duration crosses shutdown_deadline: clip or stop append warning item and hard shutdown item
Deterministic, not predictive The function always returns the same schedule for the same inputs because it never asks the wall clock for “now”.
Pause shifts the future The daemon path mutates the scheduled start times after resume, but the absolute shutdown deadline stays fixed.
Breaks are first-class Stretch and gym are real schedule entries, not UI reminders or post-processing notes.
Cursor The cursor moves forward through tasks and interruptions, which keeps the schedule monotonic.
Accumulated work The stretch insertion rule keys off work time accrued, not raw task count.
Deadline clamp The final shutdown item is always appended when hard shutdown is enabled, even if no tasks remain.

Runtime loop and state machine

`daemon.py` runs a manual Qt event loop, polls the control socket on each tick, and keeps the session inside a small set of explicit states. That structure keeps pause, resume, give-up, and finish logic from leaking into hidden globals.

idle The daemon has not started a session yet.
awaiting_task_start The blocker is visible and waiting for task confirmation.
task_active A task timer is running and the task row has been activated.
awaiting_break_start The next break is queued but not confirmed yet.
break_active Stretch or gym is active under the same lifecycle.
paused Future schedule items are frozen until resume shifts them forward.
given_up The session is marked abandoned and the daemon exits.
finished The session ended normally or by shutdown.

Socket protocol

External controls go through a Unix domain socket at ~/.local/state/locked-in/control.sock. The server binds once, accepts a single JSON object per connection, and returns a JSON response.

pause Transitions to paused, records an interruption row, and starts the pause timer.
local only
resume Restores the previous state and shifts later items by the paused duration.
time shift
give_up Marks the session given up, closes the blocker, and stops the daemon.
3 attempts
status Returns the live state, current item title, session id, and remaining schedule length.
read only
request: { "command": "pause" } response: { "status": "paused" }

Fullscreen gate internals

Task start gate In ui.py, the task gate enables Start only when the typed text matches the task title.
Break start gate Break entries hide the text field and turn the confirm button into a break-start button.
Escape is blocked The window ignores Escape and refuses to close unless the confirm path has already fired.

Trust boundary

The repo is honest about its own limits. This is a strong fullscreen blocker, not a cryptographic kiosk lock or compositor-level session lock.
Security model The code gives local friction, not hardened OS enforcement.
Operational model It works best as a systemd user service with local desktop integration and no network dependency.

Persistence layers

The simple runtime and the daemon runtime do not share a database. They keep separate stores because their lifetimes and schemas are different.

simple_todos.dbplanner runtime
  • plans
  • plan_tasks
  • daily_sessions
  • task_runs
warden.dbdaemon runtime
  • sessions
  • tasks
  • interruptions
  • control_events
Simple store `SimpleTodoStore` self-creates tables, adds missing columns, and can migrate legacy JSON into SQLite on startup.
Daemon store `Database` uses a fixed schema with session, task, interruption, and control-event tables.

What to remember

The build is layered Entry point, config, storage, algorithm, runtime, and IPC are separated to keep the code readable.
The default runtime matters The service launches the simple path, so the shipped behavior is not the daemon even though the daemon exists.
Pure logic is the anchor The schedule builder is the easiest place to test because it is side-effect free.
